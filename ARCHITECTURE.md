# Website Architecture & Development Philosophy

This document outlines the architectural constraints, Git branching strategy, content rules, and development workflows for `mrps.in`. Adhering to these rules keeps the site lightweight, framework-independent, and optimized for long-term AI and human maintenance.

---

## 1. Overall Philosophy
The site is designed to function as a permanent digital library and archive. It must remain:
- **Lightweight:** Page sizes should be minimal, requiring minimal bandwidth.
- **Framework-Independent:** No dependence on third-party frontend runtimes or frameworks.
- **Markdown-First:** Content is authored in clean, readable Markdown (`.md`).
- **AI-Friendly:** Structured simply so LLMs and coding agents can edit, write, and deploy without breaking layout states.
- **Minimal:** Avoid design bloat, dynamic scripts, and layout animations.

---

## 2. Source of Truth & Deployment Flow
**GitHub is the single source of truth.** No content should ever be edited directly inside Cloudflare Pages or any online editor. All changes must originate from local Git commits pushed to GitHub.

```text
[Developer / AI Agent]
         │
         ▼
[Git Repository (GitHub)]
         │  (git push)
         ▼
[Cloudflare Pages Build] ── (Runs `bash build.sh`)
         │
         ▼
     [mrps.in]
```
Cloudflare Pages acts strictly as a static web server, auto-deploying every commit pushed to the designated production branch.

---

## 3. Branch Strategy & Promotion Path
To prevent experimental design layouts or incomplete blog posts from breaking the live production site, use the following clean branch promotion workflow:

```text
 [experimental]  ──►  [testing]  ──►  [main]  ──►  [Cloudflare Pages]
  (Drafts & AI)      (Integration)   (Production)      (Live Deploy)
```

1. **`main` (Production Branch):**
   - The stable, active production branch.
   - Connected directly to the custom domain `mrps.in` via Cloudflare Pages.
   - Every single commit must be verified and deployable.
2. **`testing` (Integration Branch):**
   - Used to verify compiles, test layout scales, and check browser rendering before release.
   - May be connected to Cloudflare Preview Deployments.
3. **`experimental` (Research & Prototypes):**
   - Used for design iterations, new layout concepts, draft posts, and raw AI coding experiments.
   - Can be rebased, force-pushed, or deleted at any time.

---

## 4. Technology Stack & Constraints
To ensure the codebase remains maintainable for decades without dependency rot, the project enforces a strict technology ban.

### Do NOT Introduce:
* **Frameworks:** React, Next.js, Astro, Vue, Svelte, Angular, Gatsby.
* **CSS Frameworks:** Tailwind CSS, Bootstrap, Sass/SCSS.
* **Package Managers:** npm, yarn, pnpm.
* **Bundlers:** Webpack, Vite, Turbopack, Rollup.
* **Infrastructure:** Databases, Server-Side Rendering (SSR), API route endpoints, or server runtimes.

### Allowed Stack:
* **Content:** Markdown (`content/` directory).
* **Compiler:** Pandoc (native system binary).
* **Build System:** Bash (`build.sh`).
* **Templating:** Plain HTML wrappers (`templates/`).
* **Styling:** Single CSS file (`static/css/style.css`).

---

## 5. Generated Files Constraints
- The `site/` directory is the compiled output folder generated by Pandoc.
- **Never manually edit any files inside `site/`.** All modifications will be overwritten on the next build.
- To make changes, always edit files inside `content/`, `templates/`, or `static/`, then compile the site using:
  ```bash
  bash build.sh
  ```

---

## 6. Content Rules
- **Markdown First:** Content must be authored in Markdown. Use inline HTML layout wrappers only when Markdown lacks the required structural syntax.
- **Hierarchy:** Maintain a single `<h1>` tag per page (injected automatically via page frontmatter).
- **Paragraphs:** Keep paragraphs short and focused on a single point.
- **Lists:** Use bullet points or numbered lists instead of long blocks of text to improve visual scannability.

---

## 7. Homepage Philosophy
The homepage remains strictly minimalist. Its purpose is to answer three core questions:
1. **Who am I?** (Name, short bio/tagline).
2. **Where can visitors find me?** (Direct links to Instagram and GitHub).
3. **Where should they go next?** (Header/Footer navigation to Posts, GitHub Projects, and Photos).

Avoid:
- Large hero images or graphic sliders.
- Load animations or micro-interaction layouts.
- Long biographies or redundant details.

---

## 8. Media Policy
The repository must remain fast to clone and cheap to host.
- **Photos:** Never store heavy photo galleries inside the Git repository. Upload images to external storage systems (like Google Drive) and link to them from the website.
- **Videos:** Embed or link to third-party streaming hosts (like YouTube).
- **Downloads:** Use GitHub Releases when hosting large binaries or software installers.

---

## 9. Performance & Accessibility
- **Zero-Dependency Page Speed:** Deliver minimal HTML sizes, zero client-side scripts, and extremely fast load times.
- **Accessibility:** Use semantic elements (`<header>`, `<main>`, `<nav>`) to ensure readability on screen readers and compatibility with keyboard navigation.
